# NovaKV 研发日志：RESP Encoder 实现与协议校准

## 1. 实现目标

- 构建标准 RESP (REdis Serialization Protocol) 编码器。
- 支持核心响应类型：Simple String, Error, Integer, Bulk String, Null, Array。
- 确保编码过程具备二进制安全性且无多余内存拷贝。

## 2. 关键协议陷阱与修正

### 2.1 长度字段的定义解析

- **陷阱**：在 `BulkString` (`$`) 和 `Array` (`*`) 的头部误填了整行 RESP 报文的字节总数。
- **校准**：
    - `BulkString` 的长度字段必须仅代表**有效载荷 (Payload)** 的字节数。
    - `Array` 的长度字段必须代表**子元素的个数**。
- **后果**：错误的长度会导致 Redis 标准客户端（如 `redis-cli`）在解析时发生协议偏移或永久阻塞。

### 2.2 二进制安全性 (Binary Safety)

- **分析**：`SimpleString` (`+data\r\n`) 依赖于末尾的 `\r\n` 作为定界符，若 `data` 内部包含 `\0` 或 `\r\n`，会导致协议解析提前终止。
- **规范**：对于 KV 数据库的 `Value` 传输，必须强制使用 `BulkString` 编码。它通过显式的长度声明，确保了数据体中可以包含任何二进制字符而不干扰协议解析。

## 3. 性能优化路径

### 3.1 消除中间字符串拼接

- **原始做法**：使用 `+` 运算符拼接 `std::string`（如 `"+" + data + "\r\n"`）。
- **优化做法**：利用 `NetworkBuffer` 提供的连续写入能力，采用分步 `Append`（前缀 -> 数据体 -> 结尾符）。
- **收益**：避免了多次堆内存分配及不必要的中间拷贝。

### 3.2 自动扩容机制

- **细节**：移除了手动计算并调用 `EnsureWritableBytes` 的冗余逻辑。
- **规范**：由底层 `NetworkBuffer::Append` 统一负责空间检查与扩容，降低了编码层代码的复杂度与出错概率。

## 4. 底层库风险发现 (NetworkBuffer)

### 4.1 无符号整型溢出风险

- **代码位置**：`NetworkBuffer::ReadFromFd`。
- **风险点**：将 `readv` 返回的 `ssize_t`（有符号）直接接收为 `size_t`（无符号）。
- **严重性**：当系统调用出错返回 `-1` 时，接收变量会变为极大值（4,294,967,295），导致后续的缓冲区边界检查彻底失效。

## 5. 单元测试验证点

- **二进制数据校验**：验证包含 `\0` 字符的 9 字节 RESP 报文（`$3\r\na\0b\r\n`）在 `NetworkBuffer` 中的完整性。
- **Null 类型校验**：明确 `$-1\r\n` 作为 Key 不存在时的标准返回。
- **Array 递归校验**：验证数组嵌套 BulkString 时报文结构的正确性。
