## 2. 这次真正做成了什么

### 2.1 单线程闭环已经成立

网络主链路已经能跑通：

```text
epoll_wait
  -> HandleConnectionEvent
  -> HandleRead
  -> ReadFromFd
  -> RESPParser::Parse
  -> CommandExecutor::Execute
  -> output_buffer
  -> HandleWrite
  -> write(socket)
```

这一步已经通过 `redis-cli` 手动验证过：

- `SET`
- `GET`
- `DEL`
- `RSCAN`

### 2.2 线程池已经接进主链路

在线程池版本中，主链路变成：

```text
epoll_wait
  -> HandleRead
  -> RESPParser::Parse
  -> thread_pool_.enqueue
  -> worker 执行 CommandExecutor
  -> CompletedTask
  -> wake_fd_
  -> HandleWorkerCompletions
  -> output_buffer
  -> HandleWrite
  -> write(socket)
```

也就是说：

- IO 线程只负责收发、解析、事件分发、真正写 socket
- worker 线程只负责命令执行和响应生成
- worker 不直接碰 socket

### 2.3 异步回写的最小安全保护已经补上

为了解决线程池接入后的正确性问题，这次补了两层保护：

- `generation`
  - 解决 fd 复用后的错误回写
- `seq + pending_responses`
  - 解决同一连接多条异步命令完成顺序不一致的问题

现在同一连接上的异步响应会先按 `seq` 暂存，再按 `next_write_seq` 连续挂回 `output_buffer`。

### 2.4 停机顺序已经从“随便关”改成“有明确边界”

这次把停机逻辑拆成了两段：

- `Stop()`
  - 只负责发停止请求，并唤醒事件循环
- `Cleanup()`
  - 真正回收监听 fd、线程池、完成队列、连接 fd、`wake_fd_`、`epoll_fd_`

同时给 `ThreadPool` 增加了显式 `Shutdown()`，避免析构顺序全靠运气。

## 3. 这次问清楚的关键问题

### 3.1 `RSCAN` 的输出为什么是 `key value key value`

在 `redis-cli` 里看到的：

```text
1) "a"
2) "1"
3) "b"
4) "2"
```

这是对的，不是 bug。

原因是当前 `RSCAN` 的语义就是返回扁平 KV 数组：

```text
(key1, value1, key2, value2, ...)
```

这样做的原因：

- 当前 `RESPEncoder::EncodeArray` 直接可复用
- 不需要引入嵌套数组编码
- 和 Redis 里部分“扁平交替数组”风格是一致的

当前阶段不需要为了“显示更像 pair”去改协议层。

### 3.2 多个终端连接是不是“一连接一线程”

不是。

当前模型是：

- 一个服务端进程
- 一个 IO 线程跑 `epoll`
- 所有连接都挂在同一个 `epoll` 上
- 同一个 `DBImpl` 被所有连接共享

所以：

- 终端 A `SET a 1`
- 终端 B `GET a`

能看到同一份数据，是因为它们共享同一个服务端和同一个 DB 实例，不是因为“每个连接开了一个新线程”。

### 3.3 线程池和现在最大的区别是什么

这次把这个问题彻底讲清楚了：

- 单线程版的问题不是“线程数少”
- 而是 IO 线程同时做了收发、解析、执行命令

因此一条慢命令，比如较重的 `RSCAN`，会直接拖住别的连接。

线程池接入后的核心价值是：

- IO 线程保持轻量
- 命令执行从 IO 线程里拆出去
- 多连接下不会因为一条慢命令把所有请求都卡死在同一个执行栈里

### 3.4 `generation` 是干什么的

`generation` 不是装饰字段，而是线程池阶段的最小生命周期保护。

它解决的问题是：

1. 旧连接 `fd=7` 投递了 worker 任务
2. 旧连接关闭
3. 新连接又拿到了同一个 `fd=7`
4. 旧 worker 这时才完成

如果只靠 `fd` 回写，旧响应就会串到新连接上。

所以现在连接身份不是单独的 `fd`，而是：

```text
(fd, generation)
```

### 3.5 `wake_fd_` 和完成队列是干什么的

这是线程池接入里最重要的边界。

worker 完成后不能直接改连接或直接写 socket，所以要分两步：

1. worker 把结果推到 `completed_queue_`
2. worker 向 `wake_fd_` 写一个计数，唤醒 `epoll_wait`

然后 IO 线程在 `HandleWorkerCompletions()` 里统一做：

- 校验 `fd + generation`
- 处理顺序队列
- 把响应挂回 `output_buffer`
- 开启 `EPOLLOUT`

## 4. 这次踩到的坑

### 4.1 `Connection` 构造函数一变，`try_emplace` 会直接报错

一开始 `Connection` 只有 `fd`，后来加了 `generation`，结果：

```cpp
connections_.try_emplace(client_fd, client_fd);
```

直接不成立了。

正确做法变成：

```cpp
connections_.try_emplace(client_fd, client_fd, next_generation_++);
```

这个坑的本质是：`try_emplace` 的 value 构造参数必须和 `Connection` 构造函数同步。

### 4.2 `thread_pool_` 不能默认构造

`ThreadPool` 只有带线程数的构造函数，没有默认构造。

所以 `TcpServer` 一旦加上：

```cpp
Ayu::ThreadPool thread_pool_;
```

构造函数就必须显式初始化，比如：

```cpp
TcpServer::TcpServer(DBImpl* db) : executor_(db), thread_pool_(8) {}
```

### 4.3 `InitWakeFd()` 不能写成 `void`

一开始只是想“把 `eventfd` 建起来”，但很快就发现这不只是初始化动作，它还涉及：

- `eventfd()` 可能失败
- `AddEpollEvent()` 可能失败

如果它不返回 `bool`，`Start()` 就没法统一失败回滚。

### 4.4 `HandleWorkerCompletions()` 只声明不定义会直接卡在链接阶段

这是加 `wake_fd_` 后非常典型的半成品状态：

- `Run()` 里已经分支到 `HandleWorkerCompletions()`
- 但函数还没真正实现

这种时候代码结构看着像“接好了”，其实离能工作还差最后一步。

### 4.5 线程池成员析构顺序不是小事

如果 `thread_pool_` 的析构晚于：

- `completed_mu_`
- `completed_queue_`
- `wake_fd_`

那就可能出现：

- worker 线程还活着
- 但它要用的锁、队列、唤醒 fd 已经先没了

所以这次明确把 `thread_pool_` 放到这些依赖成员后面，让它先析构。

### 4.6 `HandleRead()` 进入线程池后，旧的 `output_buffer -> EPOLLOUT` 逻辑要删

单线程阶段里：

- `HandleRead()` 直接 `Execute`
- 所以它自己可以判断 `output_buffer` 是否有内容，并开启 `EPOLLOUT`

线程池阶段里：

- `HandleRead()` 只是投递任务
- 真正 append 响应的是 `HandleWorkerCompletions()`

所以单线程时代遗留的这段逻辑必须删掉，不然职责边界会乱。

### 4.7 worker 唤醒 `wake_fd_` 时不能无视返回值

如果：

- `completed_queue_` 已经推入完成结果
- 但 `write(wake_fd_, ...)` 失败

IO 线程就可能还睡在 `epoll_wait` 里，导致响应卡住。

当前最小修法是：

- `EINTR` 重试
- 其他失败先 break

后面要补日志。

### 4.8 `Stop()` 和 `Cleanup()` 不能混成一个函数

一开始 `Stop()` 既做：

- 置 `running_ = false`
- 又直接关 fd / 清连接 / 停线程池

这会把“请求停止”和“释放资源”混在一起。

更糟的是，`epoll_wait(-1)` 还可能睡着。

所以后面改成：

- `Stop()`：只发停止请求，并通过 `wake_fd_` 唤醒事件循环
- `Cleanup()`：由退出路径统一释放资源

### 4.9 `Start()` 失败时不能再走新的 `Stop()`

`Stop()` 改成“只发停止请求”后，`Start()` 里的失败回滚如果还写：

```cpp
Stop();
```

就不会清理 `listen_fd_ / epoll_fd_ / wake_fd_`，因为这时 `running_` 还没变成 `true`。

所以后来又把 `Start()` 失败回滚改成：

```cpp
Cleanup();
```

这个坑非常典型，说明停机语义一变，初始化失败路径也必须一起检查。

### 4.10 `Stop()` 发出后，当前这一批事件不能继续处理到底

如果 `Run()` 的事件遍历里不检查 `running_`，那么即使已经请求停止：

- 当前 `ready` 这一批 events 还是会被全部处理完
- 甚至还可能继续 `accept` 新连接

所以这次还补了：

- `Run()` 的 event 循环里检查 `running_`
- `HandleAccept()` 和 `HandleConnectionEvent()` 入口也做短路

## 5. 本次代码层面的关键收口

### 5.1 线程池任务边界

当前任务对象边界已经固定为：

```text
WorkerTask:
  fd
  generation
  seq
  command
```

```text
CompletedTask:
  fd
  generation
  seq
  response
```

这套边界已经足够支撑：

- 异步执行
- 安全回写
- 同连接顺序控制

### 5.2 当前停机语义

当前停机语义可以概括成：

1. `Stop()` 把 `running_` 设为 false
2. `Stop()` 通过 `wake_fd_` 叫醒 `epoll_wait`
3. `Run()` 检测到停止请求后退出循环
4. `Cleanup()` 统一释放资源

这比一开始“谁想关就直接关 fd”稳得多。
