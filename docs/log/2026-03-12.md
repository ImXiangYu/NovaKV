### 2.2 RESP 半包问题被真正修掉了

今天最关键的一个定位结论是：之前 parser 的状态机虽然记住了“解析到哪一步”，但没有记住“已经解析出来的前几个参数”。

原来的问题路径是：

```text
第一段数据到达
  -> parser 解析出前几个 bulk 参数
  -> 返回 INCOMPLETE
  -> 已解析参数只存在本次 Parse() 的局部变量里
第二段数据到达
  -> parser 还记得状态
  -> 但前面已经解析出的命令参数丢了
```

这会导致半包场景下，第二次继续解析时命令内容不完整，甚至被误判成协议错误。

这次的修法是：

- 给 `RESPParser` 增加跨多次 `Parse()` 持续保存的 `current_command_`
- 只有在完整解析出一条命令时，才把它整体交给上层
- `Reset()` 时同步清理这部分状态

修完之后，半包和多段拼接解析才真正成立。

### 2.3 协议错误路径已经从“直接断开”收成“有顺序约束的错误回包”

之前非法 RESP 的处理过于粗糙，直接 `CloseConnection(fd)`，客户端只能看到断开，看不到协议错误。

后面又发现一个更深的问题：如果同一次读里前面几条命令已经进入 worker，后面紧跟非法 RESP，那么协议错误响应不能直接插进 `output_buffer`，否则会打乱同连接的响应顺序。

所以这次最终定下来的策略是：

- 协议错误先编码成错误响应
- 也按 `seq` 进入 `pending_responses`
- 仍然通过现有顺序回写链路进入 `output_buffer`
- 连接切到 `closing` 状态，不再继续读新请求
- 等前面合法请求的响应和协议错误都发完后，再关闭连接

这样同连接的请求-响应对应关系就没有被破坏。

### 2.4 最小版优雅关闭已经成立

这次把停机语义从“停止事件循环”收成了“开始收尾流程”。

当前停机链路是：

```text
SIGINT / SIGTERM
  -> server_main.cpp 信号处理函数
  -> TcpServer::Stop()
  -> 关闭监听 socket，停止 accept 新连接
  -> 现有连接统一进入 closing 状态
  -> worker 继续跑完已投递任务
  -> HandleWorkerCompletions / HandleWrite 继续收尾回包
  -> 连接逐步排空并关闭
  -> event loop 退出
  -> Cleanup()
```

这次有几个关键点被明确下来：

- `Stop()` 不再等价于“立刻退出”，而是“发起优雅关闭”
- `thread_pool_.Shutdown()` 仍然留在 `Cleanup()`，不提前放到 `Stop()`
- 停机后连接不再保留 `EPOLLIN`，避免继续收新请求
- 用连接排空条件和全局关闭判断，决定何时真正退出事件循环

## 4. 今天踩到的坑

### 4.1 `bash -lc` 会把 RESP 里的 `$1/$3` 当作 shell 位置参数展开

在用：

```bash
timeout 5s bash -lc "{ printf '*3\r\n$3\r\nSET\r\n$1\r\nx\r\n'; ... }"
```

这种命令测原始 RESP 时，`$3`、`$1` 会先被 shell 展开，导致发送出去的数据根本不是预期的协议字节流，最后服务端返回 `-ERR protocol error`。

这个问题不是 parser 还坏着，而是测试命令本身被 shell 篡改了。

### 4.2 `nc` 看起来“卡住”不等于服务没响应

只要服务端是长连接模型，`nc` 就可能在已经收到响应后继续保持连接等待更多数据。

所以：

- 看到输出后还没退出
- 需要手动 `Ctrl+C`

这不构成 bug，关键要看服务端有没有正确返回响应和日志链路。

### 4.3 `Stop()` 的语义不能和 `running_ = false` 直接绑定

一开始最容易犯的错误是：把 `Stop()` 理解成“立刻停止主循环”。

但对当前模型来说，一旦这么做：

- `Run()` 直接退出
- `Cleanup()` 立刻开始
- 线程池、连接、完成队列都会被回收

这会把尚未回写的响应直接截断掉。

所以后来明确了：

- `Stop()` 只负责发起关闭
- 真正结束主循环，要等连接和任务都收尾完成
