- 明确 `RSCAN` 的命令语义，避免与 Redis 原生 `SCAN` 混淆。

  - 未知命令与参数错误返回

## 3. 关键设计结论

- `CommandExecutor` 只负责命令语义裁决、调用 `DBImpl` 和 RESP 编码，不直接参与 socket 生命周期管理。
- `Connection` 后续只负责连接状态、输入输出缓冲和 parser，不直接承载 DB 逻辑。
- worker 线程未来只做命令执行和响应生成，socket 写回仍由 IO 线程负责。
- `RSCAN` 保持“从 `start_key` 开始返回所有可见 KV”的范围扫描语义，不追求兼容 Redis 原生 cursor-based `SCAN`。

## 4. 本日取舍与原因

- 没有继续使用 `SCAN` 这个命令名，因为它会让人误以为要兼容 Redis 原生分页扫描语义。
- 选择 `RSCAN`，是为了保留“范围扫描”的表达，同时输入更短，便于 `redis-cli` 调用。
- `RSCAN` 返回扁平 KV 数组，而不是嵌套数组或 cursor 结构，原因是当前实现成本最低，且与现有 `RESPEncoder` 能力直接匹配。

## 6. 复盘要点

- 网络层实现时，优先固定模块边界，再补 socket/epoll 细节，会比直接写 Reactor 主循环更稳。
- 自定义命令并不妨碍使用 `redis-cli` 联调，关键在于 RESP 协议兼容，而不是命令名必须完全等于 Redis 官方集合。
- 对当前项目而言，范围扫描能力比硬做 Redis 原生 `SCAN` 更有收益，也更符合有序 KV 的展示重点。
