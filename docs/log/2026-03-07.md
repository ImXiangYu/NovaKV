# NovaKV 技术复盘：RESP 解析器的“避坑与排雷” (2026-03-07)

## 1. 核心目标：构建高可用的非阻塞协议解析器

今天的任务是实现 `RESPParser`。在 C++ 网络编程中，解析器不仅仅是字符串处理，更考验对**内存安全、异常安全和 IO 边界控制**的理解。

---

## 2. 深度踩坑记录与解决方案

### 坑位 1：无处不在的“死循环” (Buffer Index Sync)

* **现象**：在处理空数组（如 `*0\r\n`）或非法协议首字节时，解析器陷入死循环，CPU 占用率飙升。
* **根因分析**：在 `EXPECT_ARRAY_SIZE` 状态下，虽然逻辑上判断出数组长度为 0 并返回了 `SUCCESS`，但*
  *没有移动 `NetworkBuffer` 的读指针**。下次 `Parse` 调用时，解析器依然从原来的位置开始读，导致无限重复解析同一段数据。
* **修复**：在任何 `return` 之前（无论是成功还是错误），只要确定当前行已解析完毕，必须立即调用 `buffer->Retrieve(len)`
  消费掉缓冲区数据。
* **复盘感悟**：网络缓冲区设计中，**“解析逻辑”与“索引移动”必须原子化同步**，否则会导致严重的逻辑粘滞。

### 坑位 2：有符号与无符号的“致命碰撞”

* **现象**：当解析 Redis 的 Null Bulk String (`$-1\r\n`) 时，长度校验失效，程序逻辑混乱。
* **根因分析**：最初 `array_size_` 和 `bulk_len_` 使用了 `size_t` (unsigned)。当遇到 `-1` 时，`size_t`
  会发生溢出，变成一个巨大的正数（2^64 - 1）。
* **修复**：将所有协议长度相关的变量统一改为 `int64_t`。
* **复盘感悟**：在处理外部协议（如 RESP, HTTP Content-Length）时，**必须考虑负数代表的语义（如 Null 或 Chunked）**，不能盲目使用
  `size_t`。

### 坑位 3：`std::stoi` 的潜在“炸弹” (Exception Safety)

* **现象**：如果客户端发送恶意数据（如 `*abc\r\n`），服务端进程直接崩溃退出。
* **根因分析**：`std::stoi` 在遇到非数字字符时会抛出 `std::invalid_argument` 异常。在高性能网络服务器中，**“未捕获的异常”是致命的
  **。
* **修复**：改用 C++17 的 **`std::from_chars`**。它不抛出异常，通过返回码 `std::errc` 告知成功与否，且不需要创建临时
  `std::string` 对象，性能极高（Zero-copy 友好）。
* **复盘感悟**：编写底层基础设施（如数据库、网络库）时，**应尽量避免使用异常处理流控制**，尤其是在处理不可信的客户端输入时。

### 坑位 4：状态机的“断电续传”与重置 (Pipelining)

* **现象**：连续发送两个命令（Pipelining）时，第二个命令解析失败。
* **根因分析**：解析完第一个完整数组后，`state_` 依然停留在 `EXPECT_BULK_DATA` 或者是上一次的状态。由于没有手动 `Reset()`
  ，下一次解析会直接错位。
* **修复**：在 `out_command.size() == array_size_` 触发 `return SUCCESS` 之前，强制调用 `Reset()`。
* **复盘感悟**：状态机不仅要能处理“数据没收全”的挂起，还要能处理“数据发多了”的收尾。**每一轮成功的解析都必须是一个干净的循环起点
  **。
