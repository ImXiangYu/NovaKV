# 新连接接入

## 1. 它在解决什么问题

`HandleAccept()` 处理的不是客户端命令，而是“有新的客户端连进来了”这件事。

这里要先区分两个 fd：

- `listen_fd`：监听 socket，只负责接收新的连接请求
- `client_fd`：真正和某个客户端通信的连接 socket

当 `epoll` 告诉我们 `listen_fd` 可读时，含义不是“有命令数据到了”，而是“有新连接可以接收了”。

## 2. `HandleAccept()` 的目标

这个函数的目标很单纯：

1. 从监听 socket 中接出新的客户端连接
2. 把新连接设置成 non-blocking
3. 为它创建 `Connection` 上下文
4. 把这个新连接注册进 `epoll`

完成这一步之后，后续 `EPOLLIN` 事件才会落到真正的 `client_fd` 上。

## 3. 为什么要单独有这个函数

在 Reactor 模型里，监听 fd 和连接 fd 的职责完全不同：

- `listen_fd`：只负责“接人”
- `client_fd`：负责读写数据

所以事件循环里必须把它们分开处理：

- 如果事件来自 `listen_fd`，走 `HandleAccept()`
- 如果事件来自某个 `client_fd`，走 `HandleConnectionEvent()`

这样职责边界才清楚。

## 4. 为什么不是只 `accept` 一次

标准写法通常是：

```cpp
while (true) {
  int client_fd = accept(...);
  ...
}
```

原因是：

- 一次 `listen_fd` 就绪，不一定只对应一个新连接
- 内核 backlog 队列里可能已经堆了多个待接入连接

如果你只 `accept` 一次，就可能把剩余连接留在队列里，下一轮事件处理会变得拖沓。

所以更正确的做法是：

- 循环 `accept`
- 直到返回 `EAGAIN` 或 `EWOULDBLOCK`

这表示当前没有更多连接可接了。

## 5. 为什么监听 fd 必须是 non-blocking

`HandleAccept()` 之所以能写成循环 `accept`，前提是监听 fd 已经被设成 non-blocking。

否则一旦队列取空，下一次 `accept` 就会阻塞住，整个 `epoll` 事件循环都会停下来。

所以这里 again 强调一次：

- `listen_fd` 必须 non-blocking
- 后面 `client_fd` 也必须 non-blocking

这不是优化项，而是事件驱动模型的前提条件。

## 6. 为什么 `accept` 还要带客户端地址参数

常见写法是：

```cpp
sockaddr_in client_addr{};
socklen_t client_len = sizeof(client_addr);

int client_fd = accept(listen_fd_,
                       reinterpret_cast<sockaddr*>(&client_addr),
                       &client_len);
```

作用是：

- 内核会把客户端地址信息填进 `client_addr`
- 即使暂时不打印日志，也建议保留这套标准写法

这样后面如果你想加：

- 客户端 IP
- 客户端端口
- 连接建立日志

就不需要再返工接口。

## 7. `accept` 返回错误时分别代表什么

核心分支通常是：

```cpp
if (client_fd < 0) {
  if (errno == EINTR) {
    continue;
  }
  if (errno == EAGAIN || errno == EWOULDBLOCK) {
    break;
  }
  break;
}
```

### 7.1 `EINTR`

表示系统调用被信号中断。

这通常不是致命错误，直接重试即可。

### 7.2 `EAGAIN / EWOULDBLOCK`

这在 non-blocking socket 上是一个“正常结束信号”。

它表示：

- 当前 backlog 里已经没有更多待接入连接
- 本轮 `HandleAccept()` 可以结束了

所以这里应该 `break`，而不是报错退出程序。

### 7.3 其他错误

这里可以先简单处理：

- 直接 `break`
- 后续再补日志

先把主链路跑通更重要。

## 8. 为什么 `client_fd` 也要再设一次 non-blocking

新连接接出来后，通常要立刻执行：

```cpp
SetNonBlocking(client_fd)
```

原因是：

- 后续这个 `client_fd` 也会被放进 `epoll`
- 后面的 `read` / `write` 都依赖 non-blocking 语义

如果 `client_fd` 还是阻塞的，那么将来某个连接卡住时，整个 IO 线程还是会被拖死。

## 9. 为什么要把连接放进 `connections_`

NovaKV 当前的设计是：

- `TcpServer` 统一管理所有在线连接
- 每个连接对应一个 `Connection` 对象

`Connection` 里保存的是这个客户端的独立上下文：

- `fd`
- `input_buffer`
- `output_buffer`
- `RESPParser`
- `closing`

所以 `HandleAccept()` 在拿到 `client_fd` 后，需要立刻为它创建一份连接上下文，并放进 `connections_`。

## 10. 为什么建议用 `try_emplace`

当前 `Connection` 的构造函数需要一个 `client_fd`：

```cpp
explicit Connection(int client_fd) : fd(client_fd) {}
```

这意味着它没有默认构造函数。

所以推荐这样写：

```cpp
auto [it, inserted] = connections_.try_emplace(client_fd, client_fd);
```

含义是：

- key：`client_fd`
- value：用同一个 `client_fd` 去构造 `Connection`

这样最自然，也最符合当前 `Connection` 的结构。

## 11. 为什么新连接注册的是 `EPOLLIN | EPOLLRDHUP`

接入后通常这样注册：

```cpp
AddEpollEvent(client_fd, EPOLLIN | EPOLLRDHUP)
```

它们分别表示：

- `EPOLLIN`：连接上有数据可读
- `EPOLLRDHUP`：对端关闭了连接的读写一侧，通常可用来识别断开

为什么当前先不加 `EPOLLOUT`：

- 现在还没做发送逻辑
- 可写事件几乎经常是“就绪”的
- 如果一上来就长期监听 `EPOLLOUT`，会产生很多没意义的唤醒

所以更合理的做法是：

- 默认只关心 `EPOLLIN`
- 等真的有待发送数据时，再动态打开 `EPOLLOUT`

## 12. 为什么中途失败要回滚

比如这条路径：

1. `accept` 成功
2. `connections_` 插入成功
3. `AddEpollEvent` 失败

如果你不回滚，就会留下一个坏状态：

- map 里以为这个连接存在
- 但 `epoll` 根本没监控它

后面连接生命周期会变得混乱。

所以正确做法是：

- `connections_.erase(...)`
- `close(client_fd)`
- 继续处理下一个连接

这就是“状态一致性”。

## 13. 对应到 NovaKV 的伪代码

```cpp
void TcpServer::HandleAccept() {
  while (true) {
    sockaddr_in client_addr{};
    socklen_t client_len = sizeof(client_addr);

    int client_fd = accept(
        listen_fd_,
        reinterpret_cast<sockaddr*>(&client_addr),
        &client_len);

    if (client_fd < 0) {
      if (errno == EINTR) {
        continue;
      }
      if (errno == EAGAIN || errno == EWOULDBLOCK) {
        break;
      }
      break;
    }

    if (!SetNonBlocking(client_fd)) {
      close(client_fd);
      continue;
    }

    auto [it, inserted] = connections_.try_emplace(client_fd, client_fd);
    if (!inserted) {
      close(client_fd);
      continue;
    }

    if (!AddEpollEvent(client_fd, EPOLLIN | EPOLLRDHUP)) {
      connections_.erase(it);
      close(client_fd);
      continue;
    }
  }
}
```

## 14. 一句话总结

`HandleAccept()` 的本质是：

把 `listen_fd` 上到来的新连接批量接出来，为每个连接建立 `Connection` 上下文，并注册到 `epoll`，让它们后续能够进入真正的读写事件处理流程。

## 15. 面试表达版本

如果面试时要简短回答，可以说：

“当监听 fd 在 `epoll` 中变成可读时，我不会把它当成普通读事件，而是进入 `HandleAccept` 循环调用 `accept`，直到 `EAGAIN` 为止。每拿到一个 `client_fd`，都会先设置成 non-blocking，再创建连接上下文，并注册 `EPOLLIN | EPOLLRDHUP` 到 epoll。这样可以把监听 socket 和客户端连接的职责明确分开，也为后续的读写事件处理建立统一的连接管理模型。”
