# NovaKV 开发日志 - 2026-03-05

## 1. 网络层核心架构设计 (Server Architecture)

为了展示底层能力并支持高并发，我们确立了基于 `epoll` + 线程池的 Reactor 模型方案：

* **TcpServer (Reactor)**: 负责监听端口，管理 `epoll` 实例。主循环通过 `epoll_wait` 监听事件。
* **Connection (Context)**: 每个客户端连接对应一个对象。它持有该连接的 `socket`、解析状态以及最关键的 **输入/输出缓冲区 (
  NetworkBuffer)**。
* **NetworkBuffer (Byte Stream Handler)**: 处理 TCP 字节流的“粘包”与“半包”问题。
* **ThreadPool (Worker)**: 将耗时的 DB 操作（如 `SET/GET`）从 IO 线程中解耦，丢给线程池执行，确保 IO 线程始终能快速响应。
* **RESP Parser**: 协议解析层，从 `NetworkBuffer` 中读取字节流并转换为结构化的 Redis 命令。

## 2. NetworkBuffer 开发要点与复盘

在实现 `NetworkBuffer` 时，我们经历了从基础设计到高性能优化的过程，解决了以下关键问题：

### A. 接口与封装问题

* **构造函数访问限权**: 在 C++ `class` 中，成员默认是 `private` 的。最初设计的构造函数无法在外部访问，已修正。
* **指针 vs 偏移量**: `Peek()` 最初返回 `size_t`，但在解析器场景下，返回 `const char*` 指针能显著简化逻辑（如直接调用
  `strncmp` 或 `std::search`）。

### B. 空间管理逻辑 (The Soul of Buffer)

* **两指针模型**: 采用 `read_index_` 和 `write_index_` 分别追踪解析和写入进度。
* **MakeSpace 策略**:
    * **搬移 (Move)**: 当缓冲区前面有空余空间时，优先使用 `std::copy` 将未读数据搬移到开头，避免不必要的 `resize`（内存申请）。
    * **扩容 (Resize)**: 仅在搬移后空间仍不足时才调用 `vector::resize`。

### C. 高性能 IO 优化 (ReadFromFd)

* **分散读 (readv)**: 引入了 `readv` 系统调用。
* **栈缓冲溢出技术**: 准备一个 64KB 的临时栈数组 (`extra_buffer`)。
    * 如果 Buffer 剩余空间够，直接读入。
    * 如果不够，溢出的部分进入 `extra_buffer`，随后再 `Append` 进 Buffer（此时会触发扩容）。
    * **价值**: 每次 `epoll_wait` 唤醒后，仅需 **1次系统调用** 即可读完所有待处理数据，且减少了从内核到用户空间的内存拷贝次数。

### D. 编码陷阱 (Bug Alert)

* **iovec 初始化错误**: 在实现 `ReadFromFd` 时，最初使用了空的 `std::vector<iovec>` 并通过下标访问，导致 Segment Fault。
* **修正**: 改用固定大小的栈数组 `struct iovec iov[2]`，既安全又高效。
