# NovaKV 网络主链路设计（2026-03-09）

目标：先把 3 月 9 日要求的最小闭环钉死，避免后面 `epoll`、线程池、连接生命周期同时展开时边界混乱。

## 1. 今天确认下来的主链路接口

### 1.1 `CommandExecutor`

- 职责：只做“命令语义裁决 + 存储接口调用 + RESP 编码”。
- 输入：`std::vector<std::string>`，来自 `RESPParser` 已经裁好的命令数组。
- 输出：将 RESP 响应直接写入连接的输出缓冲（当前先写 `NetworkBuffer`）。
- 当前支持：`SET/GET/DEL/RSCAN`。
- 当前错误格式：
  - 未知命令：`-ERR unknown command '<cmd>'`
  - 参数个数错误：`-ERR wrong number of arguments for '<CMD>' command`

### 1.2 `Connection`

- 本轮先不写完整类实现，但接口边界先固定：
  - `fd`
  - `NetworkBuffer input_buffer`
  - `NetworkBuffer output_buffer`
  - `RESPParser parser`
  - `bool closing`
  - 后续补 `connection_id/generation`，用于 worker 回写时识别悬空连接
- 责任划分：
  - IO 线程只负责 `read -> parse -> enqueue/writeback -> write`
  - 连接对象不直接做 DB 逻辑

### 1.3 线程池任务模型

- 任务最小载荷建议：
  - `connection_id`
  - 原始命令参数 `std::vector<std::string>`
  - 必要时附带 `generation`
- 执行边界：
  - IO 线程完成 RESP 解析
  - worker 线程调用 `CommandExecutor`
  - worker 不直接操作 socket，只产出响应字节或写回连接输出缓冲的动作

### 1.4 回写路径

- 单线程阶段：
  - `RESPParser -> CommandExecutor -> connection.output_buffer`
- 线程池阶段：
  - `RESPParser -> TaskQueue -> worker(CommandExecutor) -> output_buffer -> EPOLLOUT`
- 关键约束：
  - socket 写出只能由 IO 线程做
  - worker 完成后只能“挂响应”，不能直接 `send`

## 2. 当前命令映射约定

- `SET key value` -> `DBImpl::Put(key, ValueRecord{kValue, value})` -> `+OK`
- `GET key` -> `DBImpl::Get` -> 命中返回 Bulk String，未命中返回 `$-1`
- `DEL key` -> `DBImpl::Put(key, ValueRecord{kDeletion, ""})` -> `+OK`
- `RSCAN start_key` -> `DBImpl::NewIterator()->Seek(start_key)` -> 返回扁平数组：
  - `*N\r\n$key1\r\n$value1\r\n$key2\r\n$value2...`
  - 选择这个格式是因为当前存储语义返回的是可见 KV，而不是仅 key 列表
  - 同时现有 `RESPEncoder::EncodeArray` 可以直接复用，先不引入嵌套数组编码

## 3. 为什么今天先做到这里

- `CommandExecutor` 先独立出来，可以把协议层和存储层的边界固定，后面接 `epoll` 时只管“什么时候调它”，不用再讨论“调完怎么编码”。
- 先统一 RESP 错误格式，能避免后续单线程闭环和 `redis-cli` 联调时出现一半自定义字符串、一半标准错误的情况。
- `RSCAN` 先用扁平 KV 数组，是当前最省实现成本且不丢信息的形式；如果后续要更像 Redis，可以再单独演进命令协议。

## 4. 下一步（对应 2026-03-10 / 2026-03-11）

- 写 `Server/Connection` 骨架，先把 `accept + epoll + per-connection context` 立起来。
