# RESP (Redis Serialization Protocol) 协议指南

## 1. RESP 协议概述
RESP 是 Redis 客户端与服务器通信采用的序列化协议。它是基于**行**的 ASCII 协议，每一行都以 `\r\n` (CRLF) 结尾。其设计目标是：实现简单、解析快速、人类可读。

### RESP 的 5 种基本数据类型
协议通过第一个字节来区分数据类型：

| 首字节 | 数据类型 | 示例 | 说明 |
| :--- | :--- | :--- | :--- |
| **`+`** | 简单字符串 (Simple Strings) | `+OK\r\n` | 用于状态回复，如成功操作的返回 |
| **`-`** | 错误 (Errors) | `-ERR unknown command\r\n` | 错误消息回复 |
| **`:`** | 整数 (Integers) | `:1000\r\n` | 返回计数或状态码 |
| **`$`** | 大块字符串 (Bulk Strings) | `$5\r\nhello\r\n` | 二进制安全字符串，`$5` 表示后续数据长度 |
| **`*`** | 数组 (Arrays) | `*2\r\n$3\r\nGET\r\n$1\r\na\r\n` | 客户端发送命令的标准格式 |

---

## 2. 客户端命令格式实例
当客户端发送命令（如 `SET mykey myvalue`）时，它会被封装为一个 **RESP 数组**。

**原始传输流：**
```text
*3\r\n$3\r\nSET\r\n$5\r\nmykey\r\n$7\r\nmyvalue\r\n
```

**解析结构：**
1. `*3\r\n`：表示这是一个包含 3 个元素的数组。
2. `$3\r\nSET\r\n`：第 1 个元素是长度为 3 的字符串 "SET"。
3. `$5\r\nmykey\r\n`：第 2 个元素是长度为 5 的字符串 "mykey"。
4. `$7\r\nmyvalue\r\n` : 第 3 个元素是长度为 7 的字符串 "myvalue"。

## 3. 核心实现组件

### A. RESP Parser (请求解析器)
*   **功能**：从网络缓冲区（NetworkBuffer）中提取原始字节流，并将其还原为 C++ 的结构化数据（如 `std::vector<std::string>`）。
*   **核心挑战：半包处理**。在非阻塞网络 IO 中，数据可能分批到达。解析器必须记录当前的解析进度，当缓冲区数据不足以完成当前步解析时，需保留状态并退出，等待下次 IO 事件触发后继续。

### B. Command Dispatcher (指令分发器)
*   **功能**：根据 Parser 解析出的字符串数组，识别具体的命令（如 SET/GET/DEL/RSCAN），并将其路由至存储引擎（DBImpl）的对应接口。
*   **错误处理**：对于不支持的命令或参数数量错误的请求，需生成标准错误回复。

### C. RESP Encoder (响应编码器)
*   **功能**：将存储引擎的操作结果（成功状态、查询值、空值等）转换回 RESP 格式发送给客户端。
    *   **成功**：`+OK\r\n`
    *   **查询命中**：`$len\r\nvalue\r\n`
    *   **查询未命中 (NULL)**：`$-1\r\n` (Redis 特殊定义的空 Bulk String)

## 4. 为什么在解析中使用状态机？

由于网络传输的碎片化，一次 `read` 调用可能只读到了半条指令。为了实现非阻塞的高性能解析，必须采用**有限状态机 (FSM)**：

1.  **State: EXPECT_ARRAY_SIZE** —— 查找以 `*` 开头的行，提取数组长度。
2.  **State: EXPECT_BULK_SIZE** —— 查找以 `$` 开头的行，提取下一个参数的长度。
3.  **State: EXPECT_BULK_DATA** —— 根据提取的长度，读取固定字节的数据。
4.  **循环转换**：在解析完一个 Bulk Data 后，检查数组是否已满。若未满，跳回状态 2 解析下一个参数；若已满，触发命令执行。

通过状态机，解析器可以在任何时刻“暂停”并记录位置，待后续数据到达后无缝衔接，这是构建高性能网络服务器的核心逻辑。
